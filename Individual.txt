 MEMBER 1 – M KARTHIK REDDY
 ROLE: Team Leader | Federated Learning Engineer | Server Developer
================================================================================

  --- FEDERATED LEARNING SYSTEM DESIGN ---

  WHAT: Designed the full FL workflow — 3 stations, 1 server, FedAvg.
  WHY:  Traditional centralized systems share raw data, causing privacy
        leaks and bandwidth waste. FL solves this by sharing ONLY weights.
  HOW:  Defined 3-station topology (Almatti → Srisailam → Prakasam).
        Planned that each station trains independently, uploads model weights
        to a central server via TCP socket, and the server merges them.

  --- FEDAVG ALGORITHM ---

  WHAT: Implemented the Federated Averaging (FedAvg) aggregation logic.
  WHY:  Averaging weights from all stations creates a global model that
        has learned from all local data — without seeing any raw data.
  HOW:  Loads .keras models from all stations.
        For each layer: new_weight = mean(w1, w2, w3) using numpy.
        Saves merged model as aggregated_model.keras.
        Validates global model → reports Accuracy, MSE, R².

  --- TCP SOCKET SERVER ---

  WHAT: Built the server that listens for model uploads on port 2222.
  WHY:  Sockets allow fast, direct communication between clients and server
        without needing a web framework or cloud service.
  HOW:  socket.AF_INET + SOCK_STREAM → opens TCP listener.
        Each incoming connection spawns a new daemon thread.
        Uses <EOF> delimiter to detect full JSON payload receipt.
        Decodes Base64 model bytes → saves .keras file to disk.

  --- CENTRAL SERVER UI (Server.py) ---

  WHAT: Built the complete server application with 7-tab dashboard.
  WHY:  A visual dashboard lets the operator monitor all 3 stations,
        trigger aggregation, view flood alerts, and export reports.
  HOW:  CustomTkinter Dark Mode, 7 tabs (Dashboard, Station 1/2/3,
        Aggregation, History, Network Map).
        Real-time CPU+RAM monitor via psutil every 2 seconds.
        Animated splash screen with loading progress messages.

  --- PROJECT MANAGEMENT ---

  WHAT: Led the team, testing, GitHub, and documentation.
  WHY:  As team leader, ensuring everyone's work integrates correctly
        is as important as writing code.
  HOW:  Weekly meetings, task tracking, end-to-end test runs,
        GitHub repo maintenance, authored Project_Documentation.md.

 MEMBER 2 – T SIVA REDDY
 ROLE: Client App Developer | Neural Network Model Designer
================================================================================

  --- CLIENT APPLICATION (Main.py) ---

  WHAT: Built the full Tkinter client app with 4-phase workflow.
  WHY:  Each station operator needs a simple, guided interface to upload
        data, train models, and send weights to the server.
  HOW:  4 LabelFrame phases in a scrollable sidebar.
        Canvas + Scrollbar for scroll support with mousewheel.
        PIL-based icon buttons (load_icon utility).
        Animated splash screen using ttk.Progressbar.

  --- FFNN MODEL ---

  WHAT: Designed and trained the Feedforward Neural Network (baseline).
  WHY:  FFNN as a baseline lets us compare whether CNN2D actually improves
        accuracy. It is simpler, faster to train, good for linear patterns.
  HOW:  Dense(32) → Dense(16, ReLU) → Dense(1) | Adam + MSE
        100 epochs, batch=8, 80/20 split.
        Training in background thread using threading.Thread.
        queue.Queue sends epoch logs safely to the main UI thread.
        ModelCheckpoint saves best model to model/ff_weights.keras.
        Training history saved to ff_history.json for MSE graph replay.

  --- CNN-2D MODEL ---

  WHAT: Designed and trained the CNN-2D model (extension/proposed method).
  WHY:  CNN2D can detect spatial patterns in the monthly grid of rainfall
        data. It outperforms FFNN because rainfall has seasonal structure.
  HOW:  Input reshaped from (N,12) to (N,12,1,1) for 2D convolution.
        Conv2D(32,(3,1)) → MaxPool → Conv2D(64,(3,1)) → MaxPool
        → Flatten → Dense(100) → Dense(1)
        150 epochs, batch=8. Saves to extension_weights.keras.
        History saved to cnn_history.json.

  --- MODEL UPLOAD ---

  WHAT: Sends trained model to the central server.
  WHY:  The whole FL system only works if clients can reliably upload their
        model weights. This is the "communication" part of FL.
  HOW:  Opens .keras file → reads bytes → encodes as Base64.
        Creates TCP socket → connects to localhost:2222.
        Sends JSON payload + <EOF> delimiter.
        Uploads FFNN and CNN models with 1-second gap between each.

 MEMBER 3 – I SUSHMA MEGHANA
 ROLE: Data Engineer | Preprocessing & Metrics Specialist
================================================================================

  --- DATASET PREPARATION ---

  WHAT: Prepared all 6 CSV files used throughout the system.
  WHY:  ML models are only as good as the data they train on. Clean,
        well-structured station-specific data is critical for FL to work.
  HOW:  Collected monthly rainfall + annual water level data for
        Almatti, Srisailam, and Prakasam from historical records.
        Format: [SUBDIVISION, YEAR, JAN-DEC (12 cols), water_level]
        15 columns total. Missing values filled with 0.
        Split into Station_1/2/3.csv for federated station assignment.

  --- PREPROCESSING PIPELINE ---

  WHAT: Built MinMaxScaler normalization and time-series split.
  WHY:  Neural networks train faster and better on [0,1] scaled data.
        shuffle=False is critical: time-series data has temporal order.
        Shuffling would mix past and future — causing data leakage.
  HOW:  norm1 = MinMaxScaler → fits on X (12 monthly features).
        norm2 = MinMaxScaler → fits on Y (water_level).
        train_test_split(shuffle=False) → 80% train, 20% test.

  --- METRICS ENGINE ---

  WHAT: Implemented calculateMetrics() for model evaluation.
  WHY:  Without proper metrics, we can't know if our model is good.
        Inverse transform is needed to report results in real units
        (actual water levels, not normalized 0-1 values).
  HOW:  norm2.inverse_transform() → convert predicted values back.
        MSE  = mean_squared_error(true, pred)
        RMSE = sqrt(MSE)
        R²   = r2_score(true, pred)
        Accuracy = 100 - (RMSE / mean(true) x 100)
        Server uses MAPE: accuracy = 100 - mean(|true-pred|/true)*100

  --- COMPARISON BAR CHART ---

  WHAT: Built the FFNN vs CNN-2D performance comparison graph.
  WHY:  Visual comparison proves CNN2D outperforms FFNN — justifying
        our choice of CNN2D as the proposed/extension model.
  HOW:  Results stored in global lists [mse, rmse, accuracy].
        pandas DataFrame + pivot → matplotlib bar chart.

  --- HISTORICAL FLOOD DATABASE ---

  WHAT: Compiled 27 historical flood events (1903–2024).
  WHY:  Puts the project in real-world context and shows the danger
        level of the Krishna River Basin.
  HOW:  Researched and hard-coded events into Server.py History tab.
        Notable: 1977 Diviseema (10,000+ casualties),
                 2021 Sangli (500+ villages), 2009 Kurnool (80+ dead).
